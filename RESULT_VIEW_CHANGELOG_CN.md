# P2：扫描结果 UI 性能改造说明

## 修复目标

旧版每产生一条结果都会调用 `scan_uri_filter()`：

1. 清空 `log2`；
2. 加锁遍历 `log` 全部条目（上限 2000）；
3. 在循环中读取 Host 与 8 个状态码开关；
4. 重新填充 `log2`；
5. 调用 `fireTableDataChanged()`；
6. 再次遍历 `log` 统计摘要。

这会让连续结果流在 EDT 上形成全量筛选与整表重绘，造成 Burp 卡顿。

## 新架构

新增 `ScanResultViewModel`，作为唯一的结果展示投影层：

- **原始结果仓库**：沿用原插件 `log`，不改动请求/响应保存与消息查看能力；
- **Host 索引**：`host -> List<EntryMeta>`；
- **状态码索引**：`status -> List<EntryMeta>`，另有“其他状态码”桶；
- **统计索引**：全局与每 Host 的状态码计数器；
- **视图投影**：仅把当前 Host + 状态码过滤结果的当前页写入旧 `log2`；
- **页大小**：默认 500，用户可在“可靠性 → 结果展示性能”中选择 100–1000；
- **刷新节流**：扫描线程把新条目写入无锁缓冲队列，EDT 每 **350ms** 合并一次；
- **增量通知**：普通追加使用 `fireTableRowsInserted()`；页内容变化使用 `fireTableRowsUpdated()`；只有用户切换筛选条件、切页或手动重建索引时才会调用有界页面级 `fireTableDataChanged()`。

## 兼容处理

通过字节码补丁将原私有方法替换为薄包装：

- `scan_uri_filter()` → `CgnEnhancement.scanUriFilter()`；
- `fireScanTableChanged()` → `CgnEnhancement.fireScanTableChanged()`。

扩展初始化尚未完成时，包装方法自动回退到原始实现，保证加载顺序兼容。

## 外部/遗留写入

可靠性扫描链路在 `addScanLogEntry()` 成功后立即向结果视图发布新条目，因此正常扫描不会再扫描整个日志列表。

为兼容未知的遗留写入路径，EDT 每 2.5 秒基于“最后一条对象身份”执行一次尾部协调；只有最后已索引条目被完全移除（例如用户清空日志）时才进行一次全量索引重建。该协调不在每个结果到达时执行。

## 验证

集成测试新增 650 条连续日志：

- 索引保留全部新增结果；
- 结果表默认最多投影 500 行；
- 普通追加没有触发全量索引重建；
- 状态码筛选可由索引投影得到空结果；
- 原有可靠任务、Scope、重试、主动探针与 Swing 边界测试仍通过。

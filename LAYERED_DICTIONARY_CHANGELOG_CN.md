# 分层字典扫描变更说明

## 目标

把原来的“全局平面字典”升级为“字典组 + 生效层级 + 路径拼接策略 + 关联请求模板”的分层扫描模型，减少深层目录无效探测，提高命中率和可控性。

## 新增能力

### 1. 字典组

新增配置中心页签：`分层字典`。

每个字典组支持：

- 启用 / 禁用
- 字典组名称
- 最小目录层级 / 最大目录层级
- 路径策略
- 关联请求模板
- 优先级
- 本组最大请求数
- 一行一个的字典内容

### 2. 目录层级策略

层级计算规则：

```text
/                         层级 0
/admin                    层级 1
/api/v1                   层级 2
/api/v1/user              层级 3
```

扫描时会从种子 URL 的父级目录逐层回溯，并只执行匹配当前层级的字典组。

### 3. 路径拼接策略

支持四种策略：

- 当前目录拼接：在当前观测目录下追加字典项
- 根路径拼接：始终从 `/` 下追加字典项
- 父目录拼接：在当前目录的上一级追加字典项
- 绝对路径/根路径：字典项无论是否以 `/` 开头，都按根路径处理

### 4. 关联请求模板

字典组可以绑定具体请求模板；为空时使用所有符合当前层级的启用模板。

### 5. 兼容旧字典

如果没有配置任何分层字典组，插件会自动创建只读兼容组：

```text
默认旧字典（兼容）
层级：0-4
策略：当前目录拼接
内容：原 getPathDictionaryEntries() 返回的旧字典
```

这保证升级后旧项目不会突然停止扫描。

### 6. 调度器集成

扫描任务现在按以下模型执行：

```text
种子 URL + 当前层级 + 匹配字典组 + 匹配请求模板 + Scope/黑白名单
```

最终去重 Key 增加了字典组 ID 与层级范围，避免不同字典组产生的探针互相覆盖。

### 7. 安全与资源限制

- 单个字典组最多 5000 条字典项
- 字典文件导入最大 2 MB
- 每个字典组可设置本轮最大请求数
- 导入时过滤空行、注释、CR/LF/NUL 等危险字符
- 保留前序版本的 Scope、失败重试、EDT 快照、结果分页和写操作模板拦截

## 推荐配置

### 根目录入口

```text
层级：1-1
策略：根路径拼接
字典：admin, login, api, swagger-ui, actuator
```

### API 二/三级目录

```text
层级：2-3
策略：当前目录拼接
字典：user, admin, order, config, system
```

### 敏感文件

```text
层级：1-3
策略：当前目录拼接
字典：.env, config.json, application.yml, backup.zip
```

## 注意事项

- Host、状态码和目录层级筛选仍只影响结果展示，不改变扫描范围。
- 字典组层级控制的是“该组在什么目录层级生成探针”。
- 请求模板层级控制的是“哪些模板能在该层级被使用”。两者必须同时匹配才会发请求。
- 如果模板配置了固定路径，仍遵循现有模板构造逻辑；建议对纯字典扫描模板使用空路径或通用路径策略。
